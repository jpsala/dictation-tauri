## Rules Shared By Every Alias

1. A shape alias parses its legacy request, adapts once to the canonical operation in `product-api.md`, and translates the safe result back. It never owns auth, binding, RBAC, quota, provider/model selection, privacy, or audit rules. Legacy profile draft/publish bridges are the explicit exception: they remain isolated against legacy behavior until consumer retirement and **must not** translate to, compose, or be invoked by canonical apply.
2. Runtime aliases and canonical routes use the same operation id/idempotency record and provider boundary. Profile apply aliases use the same canonical command fingerprint/receipt after their optimistic token is resolved under the authoritative lock. Comparison is provider-free; traffic is never mirrored. Accepted runtime requests make exactly one mocked/real provider call, never one per adapter; accepted apply requests make exactly one publication/revision/audit transaction, never one per adapter.
3. Handled errors are redacted JSON except a browser redirect/HTML response explicitly required by the old auth flow. No alias may preserve a legacy stack/plain-text leak.
4. Removal requires: named focused tests green; all supported consumers migrated; source/static route count zero; release/support window complete; and rollback reviewed. Removal is not part of D1.
5. Until approved cutover, production rollback is the current Cloudflare Worker. During local/canary work, a failed migration switches the whole consumer back; it never sends one request to both authorities.
6. Owners below own consumer migration and removal evidence. Product/API owns the shared adapter implementation.

## Rollback Classes

| Class | Behavior |
| --- | --- |
| `CF-DESKTOP` | Keep the supported Tauri release on the Cloudflare base URL. Revert the whole desktop adapter/base URL; never replay an ambiguous runtime operation and never mirror providers. |
| `CF-AUTH` | Route the complete browser handoff/callback plus desktop polling flow back to Cloudflare. Do not split OAuth state across authorities. |
| `CF-BFF` | Point the Control Room BFF backend to Cloudflare while its browser `/api/admin/*` contract stays unchanged. Do not dual-write mutations. |

## Atomic Apply Adaptation

The browser contract remains `POST /api/admin/profiles/apply` with its current `{ profileId, expectedActiveVersion, definition, confirmation }` body. The BFF continues to require recent Google plus publisher/owner before brokering, removes browser `actor`/`actorKey`/principal/credential fields, derives the principal/actor server-side, and keeps its backend credential server-owned.

Both the stable browser BFF adapter and Worker alias adapt legacy `profileId` to the canonical route `profileKey`. They adapt `expectedActiveVersion` to canonical `expectedRevision` only while holding the same authoritative profile lock used by apply:

1. derive a safe legacy request fingerprint from action, authenticated principal, profile ID, expected active version, canonicalized definition, and confirmation; if it already points to a committed canonical receipt, return that receipt as an idempotent replay before stale evaluation;
2. otherwise read the authoritative profile aggregate without writing;
3. require its active published version to equal `expectedActiveVersion`, otherwise return canonical `409 stale_revision` with zero writes;
4. bind the aggregate's current authoritative revision as `expectedRevision`, translate exact legacy confirmation `APPLY <profileId> v<expectedActiveVersion>` to the canonical structured confirmation, and dispatch once to the canonical apply core under that lock; the committed receipt records the safe legacy-fingerprint mapping.

The adapter cannot reinterpret the legacy version number as a revision, trust a browser revision/actor, release and reacquire the lock between mapping and commit, persist a draft, or call legacy draft/publish. Invalid profile keys, candidate definitions, and confirmations perform zero writes. An identical legacy replay maps to the committed canonical receipt and returns the same safe publication with no second version, revision, or audit. Errors are mapped to fixed JSON (`400 invalid_request`/`invalid_confirmation`, `401 unauthenticated`, `403 forbidden`, `404 not_found`, `409 stale_revision`, `422 invalid_definition`, `503 service_unavailable`) without candidate content, references, actor/principal, credentials, OAuth values, storage details, or confirmation text.

During `CF-BFF` rollback the BFF switches as a unit to the Worker alias; it never sends the same apply to both backends. The alias remains until the canonical BFF caller is deployed, the legacy backend consumer/route count is zero, focused canonical and alias tests are green, and the rollback review says the Worker route is no longer needed.
